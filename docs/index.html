EmoParse

Un sistema reproducible de análisis de emociones en discursos, construido con LLMs locales. Pensado para investigadores en lingüística, semiótica, ciencias del lenguaje y análisis del discurso que quieren procesar corpus extensos sin renunciar a la trazabilidad ni a la teoría.

01 · Qué es

Qué es EmoParse#

EmoParse toma un corpus de discursos en texto plano y devuelve, para cada frase, una caracterización semiótica de las emociones que la atraviesan: qué actor las experimenta, en qué modo de existencia, con qué foria, dominancia, intensidad, y cuál es la fuente que la desencadena.

El sistema está pensado bajo cuatro principios:

Reproducibilidad

Cada run vive en su propia base SQLite, con versionado fino de prompts, ontologías y schemas. Volver a ejecutar el mismo corpus produce los mismos resultados.

Trazabilidad

Cada emoción detectada incluye su justificación textual, su fuente, y queda enlazada a la frase original. No hay caja negra.

Local-first

Funciona con modelos GGUF locales (llama.cpp) o vía LM Studio. No es obligatorio enviar datos a servicios externos. La arquitectura admite además backends de API (OpenAI, Anthropic, etc.).

Extensible

El pipeline está organizado como un DAG declarativo. Sumar un género nuevo, una source de scraping, un agente o una stage es código aislado, no parches transversales.

Qué no es EmoParse

  • No es un detector de sentimiento tipo positivo/negativo/neutro. Trabaja con una ontología semiótica de emociones (modo de existencia, foria, dominancia, intensidad, fuente) que reconstruye la configuración emocional del discurso.
  • No es un servicio cloud. Es un paquete Python que corrés en tu máquina, idealmente con GPU. El dashboard es Streamlit local.
  • No reemplaza la lectura analítica. Está pensado como herramienta de asistencia: procesa corpus grandes, expone resultados estructurados, y deja la interpretación final al investigador.
Marco teórico

Las categorías teóricas (modos de existencia, foria, actante, experienciador, fuente, etc.) provienen de tradiciones como la semiótica de Greimas, la semiótica tensiva, la teoría semio-comunicacional de Verón o el enfoque retórico-discursivo de Plantin. EmoParse no impone teorías: las operacionaliza en ontologías y heurísticas editables que viven en knowledge/.

02 · Setup

Instalación#

EmoParse se instala clonando el repositorio. Requiere Python 3.11 o superior. Para uso con GPU local conviene además tener CUDA configurado.

Requisitos previos

  • Python 3.11+
  • git
  • GPU recomendada (NVIDIA con CUDA, AMD con ROCm) para correr los modelos GGUF localmente. CPU funciona pero es muy lento para corpus grandes.
  • ~12 GB VRAM para el pipeline estándar (Qwen3-14B + Phi-4-mini). El runner descarga el modelo de VRAM al cambiar de stage, así que no se cargan simultáneamente. Para el summarizer con Qwen3-30B MoE (~18 GB en Q4_K_M) se necesita una placa más grande o configurar offload parcial a RAM, pero sino se puede usar otro modelo.

Instalación desde GitHub

# 1. Clonar el repositorio
git clone https://github.com/alexdcolman/EmoParse.git
cd EmoParse

# 2. Crear entorno virtual
python -m venv .venv
source .venv/bin/activate   # Linux/macOS
# .venv\Scripts\activate    # Windows

# 3. Instalar el paquete
pip install -e .

El comando pip install -e . hace una instalación editable (los cambios al código fuente se reflejan al instante) y registra el comando emoparse en tu PATH.

Si no querés mantener modelos locales podés usar LM Studio como backend, o configurar un backend de API remoto (OpenAI, etc.). Ver Agregar un backend.

03 · Empezar

Quickstart#

Asumiendo que ya instalaste el paquete y tenés al menos un GGUF en models/, el flujo mínimo es: configurar, preparar input, correr, explorar.

  1. Copiá el config de ejemplo.

    El repo incluye config.example.yaml con todos los campos comentados.

    cp config.example.yaml config.yaml
# Editá config.yaml: ajustá paths y aliases de modelos
  2. Prepará tu input.

    EmoParse acepta CSV o JSON. Las columnas mínimas son codigo (id único) y contenido (texto del discurso). Opcionales recomendadas: titulo, fecha, url.

    # Opción A: input propio
ls data/mi_corpus.csv

# Opción B: scrapear discursos de Casa Rosada
emoparse scrape --source casarosada \
  --out data/discursos.csv \
  --from 2024-01-01 --to 2024-12-31
  3. Corré el pipeline. 
    emoparse run \
  --config config.yaml \
  --input data/discursos.csv \
  --run-id run_2024_q1

    El pipeline crea runs/run_2024_q1.sqlite con todos los resultados.

  4. Mirá el progreso. 
    emoparse status --db runs/run_2024_q1.sqlite
  5. Explorá los resultados. 
    # Dashboard interactivo (Streamlit)
streamlit run src/emoparse/app/__main__.py

# O exportar a CSV
emoparse export --db runs/run_2024_q1.sqlite --out exports/
04 · Ejemplo completo

Ejemplo end-to-end#

Caso real: querés analizar las emociones en los discursos del presidente argentino durante el primer trimestre de 2024. De cero a dashboard.

0. Setup inicial

git clone https://github.com/alexdcolman/EmoParse.git
cd EmoParse
python -m venv .venv && source .venv/bin/activate
pip install -e ".[llamacpp,ui,scraping]"

# Bajar modelo
mkdir -p models
huggingface-cli download Qwen/Qwen3-14B-Instruct-GGUF \
  qwen3-14b-q4_k_m.gguf --local-dir ./models

1. Configurar

Copiá el template y editalo: 

cp config.example.yaml config.yaml

Editá config.yaml dejando lo esencial: 

models:
  qwen3-14b:
    backend: llama_cpp
    path: models/qwen3-14b-q4_k_m.gguf
    context_length: 8192
    n_gpu_layers: -1     # -1 = todo a GPU
    temperature: 0.6
    seed: 42

  phi4-mini:
    backend: llama_cpp
    path: models/phi-4-mini-instruct-Q4_K_M.gguf
    context_length: 4096
    n_gpu_layers: -1
    temperature: 0.0
    seed: 42

pipeline:
  stages:
    summarizer:      qwen3-14b
    metadata:        phi4-mini
    enunciation:     qwen3-14b
    actors:          qwen3-14b
    emotions:        qwen3-14b
    characterizer:   qwen3-14b
  cache_enabled: true
  max_retries: 3
  timeout_seconds: 90

paths:
  runs_dir:      runs
  models_dir:    models
  knowledge_dir: knowledge
  inputs_dir:    data

versions:
  knowledge: v1
  prompt:    v1
  ontology:  v1
  schema:    v1

notes: |
  Análisis de discursos Q1 2024. Run inicial.

2. Scrapear el corpus

emoparse scrape \
  --source casarosada \
  --out data/discursos_q1_2024.csv \
  --from 2024-01-01 \
  --to   2024-03-31

El scraper produce un CSV con las columnas obligatorias (codigo, contenido, titulo, fecha, url, fuente) y dedup por URL.

3. Correr el pipeline

emoparse run \
  --config config.yaml \
  --input  data/discursos_q1_2024.csv \
  --run-id presidencial_q1_2024 \
  --genre  discurso_presidencial

Lo que pasa internamente:

  1. Se ingieren los discursos al storage SQLite.
  2. Se chunkean en frases (porque discurso_presidencial.unit == "frase").
  3. Se corren las stages activas por defecto en orden topológico: summarizer → metadata → enunciation → actors → emotions → explode_emociones → normalize_emotions → characterizer.
  4. Cada llamada al LLM pasa por el cache. Si una stage falla, el discurso queda marcado como failed y el resto sigue.
  5. Al final se imprime un reporte con conteos por stage.

4. Reintentar lo que falló

Si alguna stage tuvo fallas (por ejemplo emotions falló en 8 frases por timeout), no hace falta recorrer todo:

# Ver qué quedó pendiente
emoparse status --db runs/presidencial_q1_2024.sqlite

# Reintentar solo los failed de la stage emotions
emoparse retry --db runs/presidencial_q1_2024.sqlite --stage emotions

# Reanudar el pipeline (saltea lo ya completado)
emoparse run --config config.yaml --input data/discursos_q1_2024.csv \
  --run-id presidencial_q1_2024

5. Activar etapas opcionales

# normalize_actors: unifica menciones de un mismo actor y detecta desconocidos
emoparse run --config config.yaml --input data/discursos_q1_2024.csv \
  --run-id presidencial_q1_2024 \
  --stages normalize_actors

# Revisar y resolver los actores no reconocidos
emoparse discoveries list   --db runs/presidencial_q1_2024.sqlite
emoparse discoveries promote --db runs/presidencial_q1_2024.sqlite --id 3 \
  --canonical-id milei_javier --display-name "Javier Milei" --tipo persona
emoparse discoveries merge   --db runs/presidencial_q1_2024.sqlite --id 7 --into milei_javier
emoparse discoveries discard --db runs/presidencial_q1_2024.sqlite --id 12
emoparse discoveries apply   --db runs/presidencial_q1_2024.sqlite --kb knowledge/actors.json

# emotions_pass2: segunda lectura de emociones con contexto del discurso
emoparse run --config config.yaml --input data/discursos_q1_2024.csv \
  --run-id presidencial_q1_2024 \
  --stages emotions_pass2

# actants: análisis actancial de cada emoción (mediador, verificadores, operador)
emoparse run --config config.yaml --input data/discursos_q1_2024.csv \
  --run-id presidencial_q1_2024 \
  --stages actants

# judge: evalúa la coherencia de las caracterizaciones
emoparse run --config config.yaml --input data/discursos_q1_2024.csv \
  --run-id presidencial_q1_2024 \
  --stages judge

6. Validar coherencia semiótica

emoparse validate --db runs/presidencial_q1_2024.sqlite
# Imprime resumen de issues detectadas por los domain validators

7. Explorar resultados

# Dashboard Streamlit
streamlit run src/emoparse/app/__main__.py

# El sidebar lista todos los runs disponibles bajo runs/
# El run activo se elige ahí. No hace falta pasar --db.

8. Exportar a CSV

emoparse export \
  --db  runs/presidencial_q1_2024.sqlite \
  --out exports/presidencial_q1_2024/

# Produce:
#   exports/presidencial_q1_2024/discursos.csv
#   exports/presidencial_q1_2024/frases.csv
#   exports/presidencial_q1_2024/emociones.csv
Por qué tres CSVs

No hay un join consolidado porque cada nivel tiene su propia granularidad. discursos.csv es un row por discurso (metadata, resumen, enunciación); frases.csv es un row por frase (texto, actores y emociones como JSON); emociones.csv es un row por emoción individual con caracterización aplanada. Cada uno es el formato natural para su tipo de análisis.

05 · Conceptos

Arquitectura#

EmoParse está organizado en capas con dependencias unidireccionales. Las capas superiores conocen a las inferiores, no al revés.

Capas de EmoParse · cada capa depende solo de las inferiores

Por debajo de todo lo anterior está storage/ con la persistencia SQLite por run, y knowledge/ con la carga de ontologías y heurísticas. config/ resuelve el YAML con validación Pydantic.

Cuatro pilares

Una DB por run
Cada run produce un archivo SQLite independiente en runs/<run_id>.sqlite. Esto da reproducibilidad, comparabilidad entre runs, y permite que el dashboard lea sin bloquear al pipeline (WAL mode).
Generación estructurada
Cada agente declara un schema Pydantic. llama.cpp usa GBNF (compilado desde el schema) que restringe el sampler a nivel token. LM Studio usa response_format: json_schema en modo estricto. En ambos casos, la salida del LLM es válida por construcción.
Cache versionado
Las respuestas se cachean en la misma DB con una clave que incluye versiones de knowledge, prompts, ontología y schema. Bumpear cualquier versión invalida selectivamente el cache. Reejecuciones idénticas son gratis.
DAG declarativo
Las stages forman un grafo dirigido acíclico explícito. Pedir solo --stages emotions automáticamente valida que actors (su dependencia) ya esté completa.

El pipeline#

Un discurso atraviesa una secuencia de etapas. Las primeras operan sobre el discurso completo; las siguientes, sobre cada frase; las últimas, sobre cada emoción individual. Algunas etapas están activas por defecto y otras son opcionales: se activan solo si el investigador las pide explícitamente.

activa por defecto
opcional
Recorrido de un discurso · línea continua = etapa activa por defecto · línea punteada = etapa opcional

Las etapas

EtapaNivelQué hacePor defecto
summarizer discurso Resumen en dos pasos: parcial por fragmento y luego global consolidado. activa
metadata discurso Tipo de discurso y ubicación geográfica (ciudad, provincia, país). activa
enunciation discurso Identifica quién habla (enunciador) y a quién se dirige (enunciatarios), con sus roles. activa
actors frase Detecta los actores (personas, colectivos, instituciones) mencionados o inferidos en cada frase. activa
normalize_actors frase Unifica las distintas formas de nombrar a un mismo actor. Resuelve correferencias dentro del discurso y vincula cada mención con una base de actores conocidos. Los actores no reconocidos quedan registrados para revisión manual. opcional
emotions frase Primera lectura de emociones por frase, usando la ontología emocional, las heurísticas y los actores ya detectados. activa
emotions_pass2 frase Segunda lectura de las mismas frases, esta vez con el contexto de las emociones anteriores del discurso. Refina la primera lectura. opcional
explode_emociones emoción Etapa sin modelo: separa la lista de emociones de cada frase en filas individuales, una por emoción. activa
normalize_emotions emoción Asigna a cada emoción un nombre canónico de la ontología, de modo que variantes equivalentes ("bronca", "enojo", "indignación") se puedan agrupar en el análisis. activa
characterizer emoción Caracteriza cada emoción en detalle: orientación afectiva, dominancia, intensidad, fuente, duración y modo en que se atribuye a quien la siente. activa
actants emoción Analiza la configuración relacional de cada emoción: qué la vehiculiza, cómo se evalúa y cómo el discurso opera sobre ella (ver más abajo). opcional
judge emoción Un segundo modelo evalúa la coherencia de las caracterizaciones y emite un veredicto con nivel de confianza. opcional
Por qué dos lecturas de emociones

La primera lectura analiza cada frase de forma aislada: así se evita que el modelo "contagie" emociones de una frase a otra por inducción del contexto. La segunda lectura, opcional, revisita las mismas frases con el contexto de las anteriores, lo que permite detectar continuidades, contrastes, escaladas y emociones inducidas. Cuál usar es decisión del investigador.

Cómo se expresa una emoción en el texto

Una emoción puede aparecer en una frase de maneras muy distintas: nombrada con un sustantivo ("la alegría del pueblo"), con un adjetivo ("estábamos furiosos"), con un verbo ("temo que…"), o sin nombrarse, inferida a partir de indicios — un razonamiento, un comportamiento ("apretó los puños"), una valoración, una descripción narrativa, o una situación que cualquiera reconocería como cargada emocionalmente. EmoParse clasifica cada emoción detectada según cuál de estas formas la expresa. Esto permite estudiar regularidades del modo de expresión emocional según el orador, el tipo de discurso o la emoción misma. Las definiciones son editables y viven en la carpeta knowledge del proyecto.

Chunking

Cada género declara una unidad: frase, parrafo o documento. El splitter se elige según esto. Para discurso presidencial usamos frase (una idea por unidad); para tuit usamos documento (no se chunkea, cada tuit es una unidad). El splitter de frases es regex-puro, respeta abreviaciones del español, combina oraciones muy cortas y subdivide oraciones muy largas por ; o ,.

Backends LLM#

EmoParse define un contrato común LLMBackend que los backends concretos implementan. Hoy hay dos: llama_cpp y lmstudio. El contrato deja la puerta abierta a otros (OpenAI, Anthropic, vLLM, etc.) — ver Agregar un backend.

llama_cpp

Modelos GGUF locales vía llama-cpp-python. Generación estructurada con GBNF compilada desde el schema Pydantic (cacheada por modelo, ya que compilar GBNF es costoso). Soporta seed fija para determinismo bit-a-bit. 

models:
  qwen3-14b:
    backend: llama_cpp
    path: models/qwen3-14b-q4_k_m.gguf
    context_length: 8192
    n_gpu_layers: -1        # -1 = todo a GPU; 0 = todo a CPU
    temperature: 0.6
    max_tokens: 2048
    seed: 42

lmstudio

Cualquier servidor OpenAI-compatible local. Útil si ya tenés LM Studio corriendo o si querés gestionar modelos desde una UI. Usa response_format: json_schema en modo strict. 

models:
  qwen-local:
    backend: lmstudio
    base_url: http://localhost:1234/v1
    model_name: qwen3-14b-instruct
    temperature: 0.0
    max_tokens: 2048
    seed: 42
    timeout: 90

Configuración mixta: usar varios backends en el mismo run

Nada impide tener modelos con backends distintos en el mismo config.yaml. Cada stage puede apuntar al alias que prefiera: 

models:
  # GGUF local pesado para stages sensibles
  qwen3-14b-local:
    backend: llama_cpp
    path: models/qwen3-14b-q4_k_m.gguf
    n_gpu_layers: -1

  # LM Studio para stages medianas
  phi4-lmstudio:
    backend: lmstudio
    base_url: http://localhost:1234/v1
    model_name: phi-4-mini-instruct

pipeline:
  stages:
    summarizer:    qwen3-14b-local      # pesado en local
    metadata:      phi4-lmstudio        # liviano en LM Studio
    enunciation:   qwen3-14b-local
    actors:        qwen3-14b-local
    emotions:      qwen3-14b-local
    characterizer: phi4-lmstudio

El runner es lo suficientemente inteligente como para descargar automáticamente el modelo actual de VRAM si la próxima stage usa otro alias incompatible. Esto permite secuenciar modelos grandes en GPUs limitadas.

¿Y usar OpenAI / Anthropic / APIs pagas?

No viene built-in, pero la arquitectura lo soporta sin cambios al core. El contrato LLMBackend es la única superficie que un backend nuevo necesita implementar. Mirá Agregar un backend para el patrón completo.

Consideración importante

Las APIs cloud no garantizan determinismo bit-a-bit aún con seed fija. Si la reproducibilidad es crítica para tu investigación, conviene usar backends locales y declarar el modelo + seed en versions.

Cache y versionado#

Toda llamada al LLM pasa por CachedBackend, que decora al backend real con un cache transparente. La clave es un hash de:

  • El alias del modelo
  • El system prompt (hash SHA-256)
  • El user prompt (hash SHA-256)
  • El qualname del schema esperado
  • La seed
  • Las cuatro versiones: knowledge, prompt, ontology, schema

Bumpear una version en config.yaml invalida selectivamente todo lo que dependa de eso:

BumpeásInvalida
knowledgeTodo (datos base del proyecto)
promptStages cuyos templates Jinja2 cambiaron
ontologyStages que consumen ontologías (emotions, characterizer)
schemaStages con schema modificado
Práctica recomendada

Bumpeá al menos la version correspondiente cada vez que toques un prompt o una ontología. Si no, el cache devuelve respuestas viejas (pre-cambio). El bumpeo es manual a propósito: te obliga a decidir qué cambio invalida qué.

Para inspeccionar el estado del cache: 

emoparse stats --db runs/<run_id>.sqlite

Imprime conteo de entradas, hit rate de la sesión, y desglose por modelo.

Géneros#

Un género es un descriptor declarativo de cómo varía el pipeline para un tipo de discurso. Cada género define:

  • Unidad de chunking: frase, parrafo o documento.
  • Roles enunciativos válidos: tupla cerrada de strings (ej. prodestinatario para discurso político vs. seguidor para tuit).
  • Overrides opcionales: modelos por stage, tamaño de batch, summarizer on/off, prompts custom.

EmoParse trae dos géneros built-in:

discurso_presidencial

Unit frase; roles prodestinatario, paradestinatario, contradestinatario (Verón). Default si no pasás --genre.

tuit

Unit documento; roles seguidor, oponente, audiencia_general. Sin summarizer. Batch sizes más grandes (10).

El género se elige con --genre <id> en el CLI. Los roles declarados se inyectan dinámicamente en el schema Pydantic EnunciatarioSchema.tipo (vía pydantic.create_model), lo que hace que el GBNF restrinja la salida del LLM exactamente al universo permitido por el género. El qualname del schema dinámico incluye el genre_id, así que el cache mantiene buckets separados por género.

Para crear un género nuevo, ver Agregar un género.

Validación en tres capas#

EmoParse valida la salida del LLM en tres niveles complementarios. Cada capa atrapa errores que las anteriores no pueden.

Capa 1 — Forma (token-level)

GBNF (en llama.cpp) o JSON Schema estricto (en LM Studio) restringen el sampler a producir salidas estructuralmente válidas. Es imposible que el LLM devuelva JSON malformado o con campos extra.

Capa 2 — DataFrame contracts (entre stages)

Cada stage valida su entrada y salida contra un contrato Pandera (DiscursoInputContract, FraseConActoresContract, EmocionExplodedContract, etc.). Esto atrapa errores de integración: una stage que produce columnas mal nombradas, tipos inesperados, índices duplicados.

Capa 3 — Coherencia semiótica (domain validators)

Los RowValidators y DiscursoValidators ejecutan reglas teóricas sobre las emociones caracterizadas. Detectan incoherencias del tipo:

IDRegla
V01Modo virtual o potencial no puede tener al enunciador como experienciador.
V02Fuente no identificada con intensidad alta es tensión semiótica.
V04Foria afórica incompatible con intensidad alta.
V05Foria ambifórica no coexiste con intensidad baja.
V06Modo virtual con foria afórica es contradictorio.
V07Tipo de fuente actor sin actor nombrado.
V08Experienciador coincide con enunciador identificado.
V09Emoción duplicada del mismo actor en la misma frase.
V10Modo potencial con experienciador que no es enunciatario.
V11La emoción detectada no coincide con ninguna entrada de la ontología emocional, o viola las restricciones declaradas para esa emoción (por ejemplo, una orientación afectiva incompatible). Requiere tener la ontología cargada.

Las incidencias se persisten en la tabla validation_issues. Se ejecutan con emoparse validate y se pueden consultar desde el dashboard. La última regla solo se activa si se le indica al comando dónde está la ontología de emociones; en caso contrario, se omite sin bloquear el resto.

Capa 3 bis — LLM-as-judge (opcional)

El stage opt-in judge usa otro LLM para evaluar la coherencia global de cada caracterización. Devuelve un veredicto (coherente: bool, issues: str, confianza: alta/media/baja) que se persiste en la tabla judgments. Complementa los domain validators con una lectura más holística.

Análisis actancial de la emoción#

Más allá de detectar qué emoción aparece y cómo se expresa, EmoParse puede analizar la configuración relacional de cada emoción: el conjunto de elementos discursivos que la rodean, la sostienen, la evalúan o intentan modificarla. Esta etapa es opcional y se ejecuta solo si el investigador la solicita.

El análisis se hace emoción por emoción, no frase por frase. Una misma frase puede sostener varias emociones con configuraciones distintas. Por ejemplo, en "el otro día lloré porque se murió mi hijo" hay una tristeza sostenida por un comportamiento ("lloré"); pero esa misma frase, dicha ante un auditorio, puede funcionar como un recurso del discurso para producir emoción en quien escucha. Son dos configuraciones diferentes para la misma frase.

Cada emoción se describe con cuatro componentes. Cada uno puede estar presente o ausente, y cada uno se acompaña de una breve justificación citando el texto.

ComponenteQué identifica
Mediador Aquello que vehiculiza la emoción entre su origen y quien la siente: el propio discurso del orador, una voz citada, un documento, un objeto, un espacio, o una acción. Cuando la emoción se vincula con su origen sin intermediario, el mediador se marca como ausente.
Verificador normativo Una operación del discurso que evalúa la emoción según una norma — cultural, moral, jurídica, ideológica o estética —, ya sea para legitimarla ("está bien indignarse con esto") o para rechazarla ("no deberías estar triste por eso").
Verificador observacional Una operación que evalúa si la emoción es auténtica o si su causa real es la declarada. Incluye poner en duda la sinceridad ("no parecías enojado") o reinterpretar el desencadenante ("lo que en realidad te molesta es otra cosa").
Operador de modificación Una operación dirigida a actuar sobre la emoción de alguien: argumentar a favor o en contra de ella, anticipar un estado emocional deseable, buscar activarla deliberadamente, o bloquearla y deslegitimarla.
Componentes configurables

Si para un corpus determinado alguno de los cuatro componentes no resulta informativo, puede desactivarse individualmente sin tocar código. El sistema lo registra entonces como ausente de forma uniforme, manteniendo estable la estructura de los resultados.

Homogeneización de actores y emociones

Un mismo actor puede aparecer nombrado de muchas formas a lo largo de un discurso ("el presidente", "Milei", "el primer mandatario"). De modo análogo, una misma emoción puede expresarse con palabras distintas ("bronca", "enojo", "indignación"). Para que el análisis agregado sea posible, EmoParse incluye dos etapas opcionales de normalización:

  • Una unifica las menciones de actores: resuelve las referencias internas del discurso y vincula cada mención con una base de actores conocidos. Las menciones que no encuentran correspondencia quedan registradas como hallazgos para revisión manual, sin modificar automáticamente la base.
  • Otra asigna a cada emoción un nombre canónico tomado de la ontología, de modo que las variantes equivalentes puedan agruparse al analizar tendencias.

Los hallazgos de actores no reconocidos se gestionan con el subcomando emoparse discoveries (listar, exportar, promover a nuevo actor, fusionar con uno existente o descartar).

--kbPath al JSON de la base de knowledge de actores. Requerido.

Flujo recomendado

list → revisar en terminal o export a CSV para lotes grandes → registrar decisiones con promote / merge / discard → apply --dry-run para confirmar → apply. Las decisiones fallidas quedan marcadas con su error y se pueden corregir registrando una nueva decisión (sobrescribe la anterior).

Agregar una source de scraping#

Una source es un adapter que sabe (a) listar URLs de discursos en una fuente y (b) extraer el contenido de cada uno. La fuente built-in es casarosada. Para agregar otra, implementás SourceAdapter.

  1. Crear el módulo del adapter.

    Bajo src/emoparse/scraping/sources/mi_fuente.py:

    from __future__ import annotations
from collections.abc import Iterator
from datetime import date

from emoparse.scraping.base import DiscursoRecord, SourceAdapter
from emoparse.scraping.http_client import HttpClient
from emoparse.scraping.normalize import clean_whitespace, normalize_date


class MiFuenteAdapter(SourceAdapter):
    source_id = "mi_fuente"
    requires_selenium = False

    def __init__(self, timeout: float = 20.0, max_retries: int = 3) -> None:
        self.http = HttpClient(timeout=timeout, max_retries=max_retries)

    def list_discursos(
        self,
        max_items: int | None = None,
        from_date: date | None = None,
        to_date: date | None = None,
    ) -> Iterator[str]:
        # Iterá URLs absolutas a discursos individuales
        ...

    def fetch_discurso(self, url: str) -> DiscursoRecord | None:
        # Descargá y parseá el discurso. Normalizá antes de devolver.
        html = self.http.get(url).text
        # ... extraer título, fecha, contenido ...
        return DiscursoRecord(
            codigo="...",
            url=url,
            titulo="...",
            fecha=normalize_date("..."),
            contenido=clean_whitespace("..."),
            fuente=self.source_id,
            extras=(),
        )

    def close(self) -> None:
        self.http.close()
  2. Registrar en el SOURCES registry.

    En src/emoparse/scraping/sources/__init__.py:

    from emoparse.scraping.sources.mi_fuente import MiFuenteAdapter

SOURCES: dict[str, type[SourceAdapter]] = {
    CasaRosadaAdapter.source_id: CasaRosadaAdapter,
    MiFuenteAdapter.source_id:   MiFuenteAdapter,   # ← nuevo
}
  3. Usarla desde el CLI. 
    emoparse scrape --source mi_fuente --out data/discursos.csv
Diseño

Los adapters no escriben a disco; devuelven DiscursoRecord. La persistencia la maneja el CLI vía CsvAppender. Esto permite testear adapters con fixtures HTML y reusarlos desde notebooks o jobs externos.

Agregar un género#

Un género nuevo se distribuye como módulo Python y se registra vía entry-points. No requiere modificar el core de EmoParse.

  1. Crear el módulo del género.

    Puede vivir en EmoParse mismo (src/emoparse/genres_examples/) o en un paquete externo.

    # mi_paquete/genres/comunicado.py
from emoparse.genres.base import Genre


def get_genre() -> Genre:
    return Genre(
        genre_id="comunicado",
        display_name="Comunicado institucional",
        unit="parrafo",
        enunciation_roles=(
            "destinatario_directo",
            "audiencia_amplia",
            "contraparte",
        ),
        models={},              # sin overrides
        batch_size={"emotions": 8},
        summarizer=True,
        prompt_overrides={},
    )
  2. Registrar el entry-point.

    En el pyproject.toml del paquete que lo expone:

    [project.entry-points."emoparse.genres"]
comunicado = "mi_paquete.genres.comunicado:get_genre"
  3. Reinstalar para registrar entry-points. 
    pip install -e .
  4. Usarlo. 
    emoparse run --config config.yaml --input data/x.csv \

Qué pasa internamente

El registry descubre el género vía importlib.metadata.entry_points. Los enunciation_roles se inyectan en un schema Pydantic dinámico que restringe la salida del LLM exactamente a esos roles. El qualname del schema dinámico incluye el genre_id, así que el cache LLM mantiene buckets separados por género — un análisis de discurso_presidencial y otro de comunicado sobre el mismo texto NO comparten hit, lo cual es correcto.

Agregar un agente#

Un agente es la unidad mínima que ejecuta un prompt sobre un dataframe. Hay dos clases base: BaseAgent (una fila por llamada) y BaseBatchAgent (varias filas por llamada, correlacionadas con unit_idx).

Estructura de un agente nuevo

# src/emoparse/agents/mi_agente.py
from __future__ import annotations
import pandas as pd

from emoparse.agents.base import BaseBatchAgent
from emoparse.core.backend import LLMBackend
from emoparse.core.schemas import MiSchemaBatch, MiBatchItem
from emoparse.core.prompts import mi_agente as tpl


class MiAgente(BaseBatchAgent):
    SCHEMA = MiSchemaBatch
    BATCH_SIZE = 5

    def __init__(self, backend: LLMBackend, titulo: str = "") -> None:
        self.titulo = titulo
        super().__init__(backend=backend)

    def _build_system(self) -> str:
        return tpl.render_system(titulo=self.titulo)

    def _build_user(self, batch: pd.DataFrame) -> str:
        unidades = "\n".join(
            f"[{idx}] {row.frase}"
            for idx, row in batch.iterrows()
        )
        return tpl.render_user(unidades_block=unidades)

    def _map_item_to_columns(
        self, item: MiBatchItem, row: pd.Series
    ) -> dict:
        return {"mi_resultado": item.valor}

Pasos completos

  1. Definir el schema Pydantic en core/schemas.py (heredar de StrictBase con extra="forbid").
  2. Crear los templates Jinja2 en core/prompts/templates/ (system + user) y un wrapper core/prompts/mi_agente.py con render_system() y render_user().
  3. Implementar la clase del agente heredando de BaseAgent o BaseBatchAgent.
  4. Para que se ejecute en el pipeline, además hay que crear una stage que lo invoque.
Buena práctica

Mantené el agente puro: recibe un LLMBackend y un DataFrame, devuelve un DataFrame con columnas nuevas. No toca DB ni filesystem. La persistencia es responsabilidad de la stage que lo usa. Eso permite testear el agente con backends fake y fixtures CSV.

Agregar una stage#

Una stage es el wrapper de orquestación que conecta un agente con el storage. Sabe cargar pendientes, ejecutar el agente, validar contratos y persistir resultados o errores.

  1. Implementar la stage. Heredá de Stage, _DiscursoStage o _FraseStage según el nivel. 
    # src/emoparse/pipeline/stages.py (o un módulo aparte)
class MiStage(_FraseStage):
    NAME = "mi_stage"
    STAGE_KEY = "mi_stage"

    def _build_agent(self, input_data, codigo) -> MiAgente:
        return MiAgente(
            backend=self.backend,
            titulo=input_data.get("titulo", ""),
        )

    def _extract_payload(self, row) -> dict:
        return {"valor": row.mi_resultado}
  2. Declararla en el DAG. Editá pipeline/dag.py agregando el nodo y sus dependencias: 
    EMOPARSE_DAG = StageDAG([
    # ... nodos existentes ...
    StageNode("mi_stage", deps=("emotions",)),
])
  3. Registrarla en el runner. Agregá una rama en PipelineRunner._build_stage() que sepa instanciar MiStage.
  4. Persistencia. Si necesita columnas nuevas en alguna tabla, editá storage/schema.py y agregá una migración aditiva en storage/runs.py:_apply_additive_migrations.
  5. Usarla. El runner la corre automáticamente cuando esté en enabled_stages. 
    emoparse run --config config.yaml --input data/x.csv \
  --run-id run --stages mi_stage

Agregar un backend#

Caso típico: querés usar la API de OpenAI, Anthropic, Groq, o un endpoint de inferencia propio. El contrato LLMBackend es la única superficie a implementar.

Patrón general

# src/emoparse/core/backend/openai_api.py
from __future__ import annotations
from typing import Any, TypeVar
import time

from pydantic import BaseModel
from openai import OpenAI, APITimeoutError, APIConnectionError

from emoparse.core.backend.base import LLMBackend, LLMResponse, TokenUsage
from emoparse.core.backend.exceptions import (
    BackendTimeoutError, BackendUnavailableError,
    ContextLengthExceededError, SchemaViolationError,
)

T = TypeVar("T", bound=BaseModel)


class OpenAIBackend(LLMBackend):
    def __init__(self, alias: str, model_config: dict[str, Any]) -> None:
        self.alias = alias
        self.model_id = model_config["model_name"]
        self.client = OpenAI(api_key=model_config["api_key"])
        self.temperature = model_config.get("temperature", 0.0)
        self.max_tokens = model_config.get("max_tokens", 2048)
        self.seed = model_config.get("seed")
        self.timeout = model_config.get("timeout", 90)

    def healthcheck(self) -> bool:
        try:
            self.client.models.retrieve(self.model_id, timeout=10)
            return True
        except Exception:
            return False

    def generate(
        self, system, user, schema=None, max_tokens=None,
        temperature=None, seed=None, stop=None, reset_before=False,
    ) -> LLMResponse:
        kwargs = {
            "model": self.model_id,
            "messages": [
                {"role": "system", "content": system},
                {"role": "user",   "content": user},
            ],
            "temperature": temperature or self.temperature,
            "max_tokens": max_tokens or self.max_tokens,
            "seed": seed or self.seed,
            "timeout": self.timeout,
        }
        if schema:
            kwargs["response_format"] = {
                "type": "json_schema",
                "json_schema": {
                    "name": schema.__name__,
                    "schema": schema.model_json_schema(),
                    "strict": True,
                },
            }
        t0 = time.perf_counter()
        try:
            r = self.client.chat.completions.create(**kwargs)
        except APITimeoutError as e:
            raise BackendTimeoutError(str(e)) from e
        except APIConnectionError as e:
            raise BackendUnavailableError(str(e)) from e

        raw = r.choices[0].message.content or ""
        parsed = schema.model_validate_json(raw) if schema else None

        return LLMResponse(
            parsed=parsed,
            raw=raw,
            usage=TokenUsage(
                prompt_tokens=r.usage.prompt_tokens,
                completion_tokens=r.usage.completion_tokens,
            ),
            latency_ms=(time.perf_counter() - t0) * 1000,
            model_alias=self.alias,
            cache_hit=False,
            finish_reason=r.choices[0].finish_reason or "stop",
        )

    def close(self) -> None: pass
    def reset_state(self) -> None: pass

Privacidad

Si trabajás con discursos sensibles o corpus académicos privados, considerá las implicaciones de enviar texto a APIs externas. Los backends locales (llama.cpp, LM Studio) garantizan que los datos no salen de tu máquina.

Errores a mapear

Convertí los errores específicos del SDK a la jerarquía de excepciones de EmoParse: BackendTimeoutError y BackendUnavailableError son transitorios y activan retry; ContextLengthExceededError, SchemaViolationError y BackendConfigError son permanentes.

Agregar un validator semiótico#

Los validators codifican reglas de coherencia teórica que el LLM debería respetar. Hay dos tipos: RowValidator (una emoción) y DiscursoValidator (todas las emociones de un discurso).

  1. Implementar la regla.

    En src/emoparse/domain/validators/rules.py:

    from emoparse.domain.validators.base import RowValidator, ValidationIssue


class V11_MiRegla(RowValidator):
    VALIDATOR_ID = "V11_mi_regla"

    def validate(
        self, codigo, frase_idx, emocion_idx,
        experienciador, tipo_emocion, modo_existencia,
        foria, dominancia, intensidad, tipo_fuente, fuente,
        enunciador, enunciatarios,
    ) -> list[ValidationIssue]:

        # Ejemplo: foria eufórica con dominancia corporal y baja intensidad
        # suele ser ruido del modelo, no un patrón válido.
        if (foria == "euforico"
                and dominancia == "corporal"
                and intensidad == "baja"):
            return [ValidationIssue(
                validator_id=self.VALIDATOR_ID,
                mensaje="Combinación inusual: eufórico + corporal + baja intensidad",
                codigo=codigo,
                frase_idx=frase_idx,
                emocion_idx=emocion_idx,
                contexto={"foria": foria, "dominancia": dominancia},
            )]
        return []

]

Decisiones de diseño#

Las decisiones de arquitectura más importantes están justificadas explícitamente. Si te preguntás por qué algo está hecho de cierta forma, probablemente la respuesta esté acá.

Una DB SQLite por run, no una compartida

Cada run produce un archivo independiente en runs/<run_id>.sqlite. Esto da reproducibilidad (cada run lleva su versión de prompts, ontologías, schemas), permite comparar runs trivialmente, y evita lock contention con el dashboard (gracias a WAL mode). La alternativa — una DB global con una columna run_id — generaría queries más complejos sin beneficios.

Sin ORM

Usamos sqlite3 de la stdlib con wrappers finitos. Un solo backend, sin migrations complejas, schemas simples: SQLAlchemy introduciría complejidad sin pagar su costo.

Pydantic + GBNF para garantizar la forma del output

Toda salida estructurada del LLM se restringe a nivel sampler: para llama.cpp compilamos GBNF desde el schema Pydantic v2; para LM Studio usamos response_format: json_schema strict. El JSON inválido es imposible por construcción. Esto saca una fuente entera de bugs.

Cache versionado con granularidad fina

Cuatro versions independientes (knowledge, prompt, ontology, schema) permiten invalidar selectivamente. Cambiar solo un prompt no invalida los datos del summarizer. Bumpear es manual a propósito: te obliga a decidir qué cambio invalida qué.

DAG declarativo en lugar de lista hardcodeada

EMOPARSE_DAG declara las dependencias de cada stage. Eso permite validar subsets habilitados (no podés correr emotions sin actors) y abre la puerta a paralelización entre stages independientes en el futuro.

Roles enunciativos restringidos por género a nivel sampler

En lugar de tener un Literal[*all_possible_roles] global (donde el modelo podría devolver "prodestinatario" en un análisis de tuit), el schema EnunciatarioSchema se genera dinámicamente con pydantic.create_model restringiendo el Literal al subset del género activo. La GBNF resultante hace que la salida inválida sea imposible.

Tres CSVs en la exportación, no un join

Cada nivel tiene granularidad propia. discursos.csv aplana payloads de stages a nivel discurso. frases.csv mantiene actores/emociones como JSON (aplanarlos explotaría el número de columnas). emociones.csv aplana la caracterización porque a ese nivel cada emoción sí tiene un set fijo de atributos.

Splitter de oraciones propio en lugar de NLTK/spaCy

El pipeline downstream necesita una idea por unidad — más granular que el splitter del summarizer. Pero NLTK/spaCy son dependencias pesadas para una tarea de regex extendido. Implementamos un splitter por oraciones con tolerancia a abreviaciones en español, combinación de oraciones cortas y subdivisión de oraciones largas. Se puede sustituir inyectando un splitter custom al Runner si un corpus específico lo requiere.

Stdlib + finitos wrappers en todo el proyecto

Donde otros agregarían un framework, EmoParse usa stdlib + un wrapper: sqlite3 en lugar de SQLAlchemy, regex extendido en lugar de spaCy, toposort propio en lugar de networkx, argparse en lugar de click. Esto baja drásticamente la superficie de mantenimiento y la cantidad de cosas que pueden romperse al actualizar dependencias.

Two-pass emotions con rolling summary

El primer pase analiza cada frase aislada para evitar alucinaciones por inducción contextual. El segundo pase (opt-in) usa contexto de las emociones anteriores del mismo discurso para detectar continuidades, contrastes y escaladas. Dos pasadas con propósitos distintos es más robusto que una sola con todo el contexto.

Validación en tres capas independientes

Forma (GBNF/JSON Schema), tipo (Pandera contracts entre stages), coherencia semiótica (domain validators + LLM-as-judge). Cada capa atrapa lo que las anteriores no pueden. Las tres son baratas comparadas con el costo del LLM.

Esquema de la base de datos#

Lectura del SQLite directamente (con sqlite3 CLI o desde un notebook) es perfectamente válida. Estas son las tablas principales.

TablaGrainContiene
runs 1 row Metadata del run: run_id, timestamps, estado, versions, config, notes.
discursos 1 row × discurso Input original (JSON) y outputs por stage a nivel discurso (summarizer, metadata, enunciation).
frases 1 row × (codigo, unit_idx) Texto de la frase y resultados de las etapas a nivel frase: actores detectados, actores normalizados, emociones de la primera y la segunda lectura.
emociones 1 row × emoción individual Una fila por emoción individual: quién la siente, tipo, modo y justificación, más el nombre canónico, la caracterización detallada y, si se ejecutó, el análisis actancial.
llm_cache 1 row × cache key Respuestas cacheadas con metadata (tokens, latencia, finish_reason, hits).
validation_issues 1 row × issue Issues detectadas por los domain validators.
judgments 1 row × emoción juzgada Veredictos del JudgeStage (coherente, issues, confianza).
run_metrics 1 row × stage × ejecución Snapshot de métricas: items OK/failed, latencias (p50, p99), tokens, cache hits/misses.

Los payloads se guardan como JSON en columnas TEXT. Esto evita migraciones cuando un agente agrega un campo: lo que cambia es el contenido del JSON, no el schema de la DB. La versión del agente se guarda en columnas separadas (summarizer_version, etc.) para auditoría.

Query ad-hoc desde sqlite3: 

sqlite3 runs/mi_run.sqlite \
  "SELECT codigo, COUNT(*) FROM frases WHERE emociones_payload IS NOT NULL GROUP BY codigo;"

Troubleshooting#

Out of memory al cargar el modelo

Bajá n_gpu_layers en el modelo afectado (de -1 a un número concreto), bajá context_length, o usá un modelo cuantizado más chico (Q4_K_S en lugar de Q4_K_M). El runner descarga automáticamente el modelo actual de VRAM cuando la próxima stage usa otro alias, así que mezclar modelos grandes funciona si se secuencian bien.

El cache no parece estar funcionando

Chequeá que cache_enabled: true esté en config.yaml. Mirá emoparse stats para ver hit rate y conteos por modelo. Si bumpeaste alguna version sin querer, el cache no encuentra hits porque la clave cambió.

Algunas frases salen con emociones: []

Es legítimo: si el modelo no detecta emociones, devuelve lista vacía. Si eso ocurre de manera sistemática en frases donde claramente hay emoción, chequeá: (1) que la ontología esté completa para el tipo de emoción, (2) que las heurísticas cubran el caso, (3) que el modelo no esté siendo sub-dimensionado para la tarea. Una opción es activar emotions_pass2 que revisita con contexto.

Errores de schema "extra fields not permitted"

Los schemas Pydantic son strict (extra="forbid"). Si un backend nuevo devuelve campos extra, los va a rechazar. Asegurate que el backend implemente correctamente strict mode (en LM Studio, strict: true en el response_format).

El dashboard no encuentra mi run

El sidebar escanea paths.runs_dir de tu config (default: ./runs). Verificá que el dashboard se lance desde el mismo directorio donde corriste el pipeline, o setea EMOPARSE_RUNS_DIR apuntando al lugar correcto.

Casa Rosada me bloquea

El adapter casarosada tiene fallback automático a Selenium si detecta bloqueo HTTP. Asegurate de tener el extra scraping_selenium instalado: 

pip install -e ".[scraping_selenium]"
